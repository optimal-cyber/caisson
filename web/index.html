The workflow, start to seal

Two commands from build host to disconnected deploy. The evidence comes with it.

operator@buildhost — caisson 
# seal a release at the source — SBOM + control evidence computed once
$ caisson package create ./my-app
  ✓ resolved 42 components · SBOM sealed
  ✓ mapped 61 NIST 800-53 controls · CMMC L2 evidence attached
  ✓ provenance signed · my-app.caisson ready

# carry it across the gap and export evidence on arrival
$ caisson deploy my-app.caisson --evidence-export
  ✓ seal verified · provenance chain intact
  ✓ images pushed to airgapped registry · workloads applied
  ✓ evidence bundle exported → ./evidence/my-app/

caisson with no arguments prints the map — and one honest promise: "If it isn't sealed, it isn't shipped."

Why Caisson vs. rolling your own

You can script tar, a registry mirror, and a folder of PDFs across the airgap. The problem is never day one — it's the assessment six months later, when nobody can prove what actually shipped.

Rolling your own

  • ✕ SBOM, control docs, and the artifact drift apart the moment they leave the pipeline.
  • ✕ Evidence is reconstructed by hand for every assessment — from memory and stale wikis.
  • ✕ "Which build is running behind the gap?" has no cryptographic answer.
  • ✕ Every program reinvents the same brittle mirror scripts.

With Caisson

  • ✓ The package is the evidence — SBOM, control mappings, and provenance are sealed inside one artifact.
  • ✓ Tamper-evident seal verified on arrival; the chain proves build host → disconnected deploy.
  • ✓ evidence export produces the assessment package from the artifact itself — not a parallel binder.
  • ✓ Complements the airgap tooling you already run instead of replacing it.

Caisson doesn't reinvent airgap transfer or Kubernetes. It wraps the OCI registries and clusters you already deploy to, and makes the compliance evidence a first-class, sealed part of the payload. See how it interoperates ›
